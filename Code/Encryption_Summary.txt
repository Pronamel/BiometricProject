SecureVote Encryption and Password Security Summary

Date: 2026-04-03

Overview
- Sensitive payloads now use application-layer envelope encryption (in addition to HTTPS/TLS).
- Voter biometric data is encrypted at rest and decrypted only when needed for matching.
- SDI lookup identity inputs are transmitted inside encrypted envelopes.
- Official passwords use Argon2 hashing and Argon2 verification.

1) Envelope Encryption (In Transit)

Pattern
- Client generates a random 32-byte DEK (AES key).
- Client encrypts entire JSON payload with AES-GCM.
- Client wraps DEK with server RSA public key (OAEP-SHA256).
- Client sends:
  - wrappedDek
  - encryptedPayload
- Server unwraps DEK with RSA private key and decrypts payload at endpoint boundary.

- Enforcement check: Server/Services/DatabaseService.cs:315

- Verify call: Server/Services/PasswordHasher.cs:27

6) Security Outcome

- Sensitive authentication, biometric, and SDI lookup payloads are protected by envelope encryption at the application layer.
- HTTPS/TLS still protects transport.
- Voter biometrics are protected at rest.
- Official passwords are not stored or compared in plaintext; Argon2 hashing/verification is used.

This delivers layered defense:
- Application-layer confidentiality for sensitive request bodies
- Transport security
- At-rest data encryption for biometrics/sensitive voter fields
- Strong password hashing for official credentials
